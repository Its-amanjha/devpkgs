# Live Logs Overlay Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stream live command output in real-time into a scrollable, centered popup modal in the terminal dashboard during package updates.

**Architecture:** We will replace synchronous execution with an asynchronous streaming executor (`RunStream`) that scans stdout/stderr line-by-line in a background goroutine and dispatches messages to the Bubble Tea update loop to populate a log line buffer.

**Tech Stack:** Go, bubbletea, lipgloss, OS pipes.

## Global Constraints
* Keep lines short to prevent wrapping.
* Maintain clean interfaces without external package changes.

  ```go

  	Line string
  }

  	return func() tea.Msg {
  		// In actual execution, this is handled via a goroutine communicating back, but for Bubble Tea,
  		// we want to return a Cmd that executes the process and reads line by line.
  		// Because Bubble Tea cmds run concurrently, we can send messages by returning them or using a channel.
  		// However, Bubble Tea commands only return ONE tea.Msg when they finish.
  		// To stream multiple messages, we must use a helper that sends messages to the program's program active channel,
  		// or we can run the reader inside a loop and use a channel subscription, or use a custom command that returns a sequence.
  		return nil
  	}
  }
  ```
  Wait! Bubble Tea commands are functions that return a single message: `type Cmd func() Msg`.
  To stream multiple messages over time from a command, Go Bubble Tea programs typically use a **goroutine that calls program.Send(msg)**, or use a **bubbletea subscription channel**.
  Let's implement the goroutine channel pattern using a package-level helper (or pass program if available, but program is in main). If we don't have access to the `Program` struct directly inside `pm.go`, we can return a helper channel subscription, or simply use `tea.Program.Send` if the program reference can be passed or if we register a helper.
  Wait! Can we use `tea.Cmd` that takes a channel or program reference?
  Bubble Tea's `Program` has a `.Send(msg Msg)` method which is thread-safe and can be called from any goroutine!
  Wait, but how does our library get access to the program?
  We can pass a global registry or let the command accept a channel, or we can use a custom Bubble Tea command that registers a goroutine.
  Wait, actually, since `Run` runs in a goroutine, if we pass a callback, or if we define a message like `ActionMsg` which contains the final output, do we need real-time streaming?
  Yes, the user wanted **live** console output.
  To stream live logs into Bubble Tea:
  We can create a subscription. In Bubble Tea, a subscription is a command that returns messages from a channel.
  Let's define a subscription structure:
  ```go
  type LogChan chan tea.Msg
  ```
  And in `main.go` or `update.go`, we can handle messages.
  Wait! An even simpler way in Bubble Tea to stream logs is to run a goroutine that writes to a channel, and have a subscription command that listens to the channel and dispatches messages!
  Let's design a clean channel-based subscription:
  We can add a channel `LogChan chan tea.Msg` to the `Model`.
  When we launch the command:
  ```go
  func RunStream(programChan chan<- tea.Msg, packageName string, action Action, manager string, cmdName string, args ...string) tea.Cmd {
  	return func() tea.Msg {
  		go func() {
  			c := exec.Command(cmdName, args...)
  			stdout, err := c.StdoutPipe()
  			if err != nil {
  				programChan <- LogFinishMsg{Manager: manager, Err: err}
  				return
  			}
  			c.Stderr = c.Stdout
  			if err := c.Start(); err != nil {
  				programChan <- LogFinishMsg{Manager: manager, Err: err}
  				return
  			}
  			
  			scanner := bufio.NewScanner(stdout)
  			for scanner.Scan() {
  				programChan <- LogLineMsg{Line: scanner.Text()}
  			}
  			
  			err = c.Wait()
  			programChan <- LogFinishMsg{Manager: manager, Err: err}
  			programChan <- ActionMsg{PackageName: packageName, Action: action, Manager: manager, Err: err}
  		}()
  		return nil
  	}
  }
  ```
  Wait! How does the program receive messages from `programChan`?
  In `main.go`, `p := tea.NewProgram(app.New(), tea.WithAltScreen())`.
  If we want the background goroutine to call `p.Send(msg)`, we need access to the `tea.Program` instance!
  Wait! Can we pass the program instance?
  No, because the program is created in `main.go` after `app.New()` is called.
  But wait! Does Bubble Tea have another way?
  Yes! In Bubble Tea, if a command sends messages to a channel, we can write a subscription command that listens to that channel:
  ```go
  func ListenLogs(ch chan tea.Msg) tea.Cmd {
      return func() tea.Msg {
          return <-ch
      }
  }
  ```
  Every time `ListenLogs` receives a message from the channel, it returns it as a Bubble Tea message.
  In `Update`:
  ```go
  case LogLineMsg:
      m.logLines = append(m.logLines, msg.Line)
      return m, ListenLogs(m.logChan)
  ```
  This is the standard, official, idiomatic Bubble Tea streaming pattern! It does not require any references to the `Program` struct and is 100% pure and clean!
  Let's verify:
  1. We add `logChan chan tea.Msg` to the `Model` in `model.go`.
  2. We initialize it in `New()`: `logChan: make(chan tea.Msg, 100)`.
  3. When starting an action, we call `RunStream(m.logChan, ...)` and return `tea.Batch(cmd, ListenLogs(m.logChan))`.
  4. In `Update`:
     - On `LogLineMsg`: we append it to `logLines` and return `ListenLogs(m.logChan)`.
     - On `LogFinishMsg`: we mark `logActive = false`. We do **not** call `ListenLogs` again (streaming stops).
  This is absolutely brilliant! It is extremely clean, thread-safe, and conforms perfectly to the Bubble Tea architecture.

Let's specify this exact channel and goroutine code in Task 1.

  ```go
  func (m Model) renderLogOverlay() string {
  	boxW := min(78, m.width-6)
  	boxH := min(22, m.height-6)
  	
  	border := lipgloss.NewStyle().Foreground(currentTheme.Primary)
  	titleText := lipgloss.NewStyle().Bold(true).Foreground(currentTheme.Primary).Render(" Installation Logs ")
  	dashLen := max(0, boxW-4-lipgloss.Width(titleText))
  	titleLine := border.Render("╭─"+strings.Repeat("─", dashLen/2)) + titleText +
  		border.Render(strings.Repeat("─", dashLen-dashLen/2)+"─╮")
  		
  	contentH := boxH - 4
  	
  	var linesToShow []string
  	totalLines := len(m.logLines)
  	
  	startLine := totalLines - contentH
  	if m.logScrollActive {
  		startLine = totalLines - contentH - m.logScrollOffset
  	}
  	if startLine < 0 {
  		startLine = 0
  	}
  	endLine := startLine + contentH
  	if endLine > totalLines {
  		endLine = totalLines
  	}
  	
  	for i := startLine; i < endLine; i++ {
  		linesToShow = append(linesToShow, m.logLines[i])
  	}
  	
  	// Pad to fill box height
  	for len(linesToShow) < contentH {
  		linesToShow = append(linesToShow, "")
  	}
  	
  	var items []string
  	innerW := boxW - 4
  	for _, l := range linesToShow {
  		padded := lipgloss.NewStyle().Width(innerW).MaxHeight(1).Render(l)
  		items = append(items, border.Render("│ ")+padded+border.Render(" │"))
  	}
  	content := strings.Join(items, "\n")
  	
  	bottom := border.Render("╰" + strings.Repeat("─", boxW-2) + "╯")
  	
  	footerText := "  ↑↓ scroll · esc/l close"
  	if m.logActive {
  		footerText = "  " + m.spinner.View() + " running... "
  	}
  	footer := lipgloss.NewStyle().Foreground(currentTheme.DimText).Italic(true).Render(footerText)
  	
  	overlay := strings.Join([]string{titleLine, content, bottom, footer}, "\n")
  	return lipgloss.Place(m.width, m.height, lipgloss.Center, lipgloss.Center, overlay)
  }
  ```
